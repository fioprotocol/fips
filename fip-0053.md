# Abstract

This FIP proposes to reallocate 50,000,000 FIO Tokens from the New User Bounties to the FIO Handles Giveaways bucket by minting 50,000,000 new FIO tokens into the designated FIO Handles Giveaways account and reducing the New User Bounties minting cap by the same amount.

## New actions

| Contract | Action | Endpoint | Description |
|---|---|---|---|
| fio.token | fipliii | None | Mint 50,000,000 FIO into FIO Handles Giveaways account and reduce New User Bounties cap. |

# Motivation

At FIO Chain genesis, several [token buckets](https://fio.net/token/distribution) were established to fund ecosystem growth. Specifically, as it relates to this FIP:
* FIO Handles Giveaways bucket: 125,000,000 FIO Tokens were allocated to the FIO Foundation with restrictions to be spent on FIO Chain fees to offer free FIO Handles to initial users. This was reduced in [FIP-21](https://github.com/fioprotocol/fips/blob/master/fip-0021.md) to 100,000,000 and reduced further in [FIP-23](https://github.com/fioprotocol/fips/blob/master/fip-0023.md) to 90,000,000. A prior reallocation under [FIP-48](https://github.com/fioprotocol/fips/blob/master/fip-0048.md) reallocated 38,505,959 FIO into the FIO Handles Giveaways from 13 permanently locked Initial Integrating Wallet accounts.
* New User Bounties bucket: 125,000,000 FIO Tokens were allocated to fund TPID rewards for new registrations. Those tokens are minted on-demand up tp the cap limit of 125,000,000 FIO Tokens. As of 2026-03-23, 48,193,945 FIO Tokens have been minted and 76,806,055 FIO Tokens remain to be minted.

As of 2026-03-23, FIO Handles Giveaways buckets hold 12,295,675 FIO Tokens, which, at current rate of free FIO Handle registrations and at current FIO Handle registration fee, will run out sometime in June 2026.

The cessation of free handles would have a severe impact on block producer compensation. Block producers earn fees from handle registrations, and free registrations currently constitute the vast majority of registration volume. Ending the free handles program would reduce block producer income by an estimated 80%, creating an existential risk to chain security if block producers can no longer justify the cost of running infrastructure.

Wallet integrators rely on free FIO Handle registrations as part of their onboarding flow. While most partners have fallback paid registration mechanisms, an abrupt end to free registrations would disrupt user onboarding and could damage partner relationships that have been cultivated over years.

The New User Bounties bucket is the most appropriate source for this reallocation. Transferring 50,000,000 FIO leaves approximately 27,000,000 FIO Tokens remaining, which is sufficient to maintain TPID rewards at the current elevated rate (50%) for a significant period. When the bounties bucket is eventually exhausted, TPID rewards will revert to the default 10% rate, which is the rate specified in existing integration partner MOUs and therefore an expected outcome.

With 50,000,000 additional FIO in the FIO Handles Giveaways bucket, the free handles program can continue for approximately 1 to 1.5 year, given the token price and registration rates remain the same. This provides substantial runway for the FIO ecosystem and preserve chain security and partner integrations in the interim. The total supply hard cap is unchanged by this FIP and remains at 1B.

# Rationale

Minting new tokens and simultaneously reducing the New User Bounties cap is the appropriate mechanism, because it achieves the economic equivalent of a direct transfer between buckets while respecting the on-chain accounting model. The New User Bounties bucket is a minting allowance, not a held balance, so tokens cannot be directly transferred from it. Instead, reducing `MAXBOUNTYTOKENSTOMINT` by 50,000,000 FIO and minting 50,000,000 new FIO into the FIO Handles Giveaways account has no impact on maximum Total Supply, which remains at 1B. Howver, when this FIP is executed, the Circulating Supply will jump by 50M.

A dedicated one-time action (`fipliii`) is the simplest and safest approach for this operation.

The re-execution guard (`fip53state` singleton) provides defense-in-depth beyond the BP MSIG authorization requirement. Even if the MSIG were to be inadvertently re-executed, the on-chain guard ensures that tokens can only be minted once. This is a critical safety property for an action that creates new token supply.

The implementation uses the `issue` action rather than `mintfio` because `mintfio` is restricted to Treasury and Foundation system accounts. The `issue` action provides the same minting capability while allowing tokens to be sent directly to the designated FIO Handles Giveaways account (`pkfbwyi2qzii`). The `issue` action also enforces the 1,000,000,000 FIO maximum supply cap, ensuring this mint cannot exceed protocol limits.

# Implementation

## Overview

1. New action name `fipliii`.
2. The new action can only be called once by eosio account (MSIG from BPs).
3. The target account and mint amount are hardcoded into the action.
4. The new action will mint 50,000,000 FIO and send them to the designated FIO Handles Giveaways account in a single call.
5. `MAXBOUNTYTOKENSTOMINT` is reduced from 125,000,000 FIO to 75,000,000 FIO at compile time.

## Action logic

1. Verify caller has eosio authority.
    * If not, report "missing required authority of eosio" exception.
2. Verify the action is executing on the fio.token contract.
    * If not, report "fipliii must be called on fio.token contract" exception.
3. Initialize the `fip53state` singleton if it does not already exist.
4. Check if the action has already been executed via `fip53state`.
    * If already executed, report "fip53 has already been executed" exception.
5. Verify the receiving account (`pkfbwyi2qzii`) exists on chain.
    * If not, report "fip53 receiving account does not exist" exception.
6. Send inline `issue` action to mint 50,000,000 FIO to the receiving account.
7. Update `fip53state` singleton to mark the action as executed.
8. Validate transaction size and report status: OK.

## Deployment

1. Deploy updated `fio.token` contract via BP MSIG using `setcode`.
2. Execute `fipliii` action via a separate BP MSIG.
